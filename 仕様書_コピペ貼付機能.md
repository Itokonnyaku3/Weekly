# 仕様書 — コピー/カット/ペースト & 複数選択（他ドキュメントツール同等化）

> 作成: 2026-05-30 / 対象: デイリーノート アウトライン（`ol-` 系）
> 目的: 既存のコピペ・複数選択を堅牢化し、リッチテキスト・表・画像を含む貼り付けを
> 他ドキュメントツール（Notion / Workflowy / Google Docs 等）と同等水準にする。
> 関連: `ARCHITECTURE.md` / `HANDOVER.md` / `CHANGELOG.md`

---

## 0. スコープと前提

- 対象は **デイリーノートのアウトライン**（`#ol-container`）。グリッド側の `eitem` は対象外。
- 画像は **GitHub アップロード必須を維持**（base64 フォールバックは今回スコープ外。ユーザー合意済み）。
- 段階実装（合意済みの優先順）:
  - **Phase 1 堅牢化**（最優先）
  - **Phase 2 リッチ貼付**（HTML）
  - **Phase 3 表貼付**（TSV / HTML 表）
  - **Phase 4 画像強化**
- 各 Phase は独立リリース可能。Phase 1 だけでも「動かない感」を解消する。

## 1. 現状（監査結果 v1.8.0 時点）

### 1-1. 既存実装の場所

| 画像貼付 | `olSetupPasteHandler` ~9204（addEventListener） | **GitHub必須**。`node.html`に`<img>` |

> `#ol-container` には **paste ハンドラが2系統**（inline `onpaste`＋`addEventListener`）同居している。

### 1-2. 「上手く機能していない」の根本原因
1. **内部クリップボードの完全一致が脆い**: 構造貼付は `clipText === _olMultiClipboard.text` の完全一致が条件（~6940）。改行コード差・他アプリ経由・再コピー等で一致が崩れると、プレーン解析に落ち **TODO状態 / html / 画像が失われる**。＝「貼れる時と貼れない時がある」。
2. **リッチ/表の貼付が無い**: `text/plain` しか読まない（`text/html` 未取得）。Word/Docs/Excel からの貼付が崩れる。
3. **不整合**: Ctrl+X が1件だと素通り、Ctrl+A（アウトライン全選択）無し、paste 2系統同居。
4. **構造の欠落**: 構造貼付時に `parentId` を削除（~6904）。サブタスクの親子がインデントのみに退化。

## 2. ターゲット設計

### 2-1. 中核方針 — 「システムクリップボード ＋ 複数MIME表現 ＋ 独自マーカー」
メモリ内の完全一致をやめ、**システムクリップボードに複数表現で書き出す**（主要ドキュメントツールと同方式）。

**コピー（Ctrl+C / Ctrl+X）時** — `navigator.clipboard.write([ClipboardItem])` で同時に:
- `text/plain`: インデント済みプレーン（外部アプリ用。現行踏襲）
- `text/html`: 人間可読な HTML（入れ子 `<ul><li>`、チェックボックス、画像、表）。**かつ先頭ラッパに独自マーカー** `data-pwt-clip="<base64(JSON)>"` を埋め込む。JSON は選択ノードのサブツリー（id除く構造・text・html・isTodo・checked・priority・images・**親子関係**等）。

**ペースト時** — `paste` イベントの `clipboardData` を**同期**で読み、優先順に判定:
1. `text/html` に `data-pwt-clip` マーカー有 → **JSONを復元し完全再構築**（TODO/画像/親子も保持）。＝自アプリ間の往復は無損失。
2. マーカー無の `text/html` 有 → **HTML→ノード変換**（Phase 2）。
3. `text/plain` 多行 → 現行のインデント解析（2sp=1段）。
4. 画像アイテム有 → アップロード（現行 `olSetupPasteHandler` 流用）。

> 読み取りは `paste` イベントから同期取得（権限不要）。書き込みは `ClipboardItem`（secure context 必須＝GitHub Pages https / localhost で可。keydown はユーザー操作と見なされ許可される）。`navigator.clipboard.write` 不可環境では `_olMultiClipboard`（内部）＋ `execCommand('copy')` をフォールバックとして残置。

### 2-2. クリップボード ペイロード形式
```html
<div data-pwt-clip="eyJ2IjoxLCJub2RlcyI6Wy4uLl19">  <!-- base64( {v:1, nodes:[...]} ) -->
  <ul>
    <li>...（人間可読・他アプリ貼付用）...</li>
  </ul>
</div>
```
- `v`: スキーマ版（将来互換）。
- `nodes`: `[{ text, html, indent, isTodo, checked, priority, color, bold, images, type, url, _rel }]`
  - `_rel`: サブツリー内の相対親子（index または相対パス）。元の `id`/`parentId` は持ち出さず、貼付時に **新ID＋新parentId** を再構築。

### 2-3. ペースト後のID/親子再構築
- 貼付先ノードの後ろ（サブツリー末尾）へ挿入（現行 `olPasteMultiClipboard` の挿入位置ロジック踏襲）。
- 各ノードに新 `id` を採番し、`_rel` から `parentId` を**復元**（現行の `delete parentId` をやめる）。
- インデントは貼付先基準で正規化（現行同様）。

## 3. Phase 別 詳細

### Phase 1 — 堅牢化（最優先）　★実装済 v1.9.0-05301615-clip-p1（2026-05-30）
**目的: 既存操作を確実に動かす。データ形式は最小変更で後方互換。**

> 実装メモ: `olEncodeClip/olDecodeClip/olBuildClipHtml/olReadClipMarker/olWriteClipboard/olCopyBlocks/olStructuredPaste` を新設。Ctrl+C/X 単体対応・Ctrl+A 全選択・`olContainerPaste` 統合（画像/マーカー/テキスト）・`olSetupPasteHandler` no-op化・`parentId` 保持。ブラウザ実機検証済。

1. **構造貼付をマーカー方式へ置換**
   - コピー/カット時に `text/html`（+マーカー）も書き出す。
   - ペースト時、まず `text/html` のマーカーを見て構造復元。`_olMultiClipboard` は二次フォールバックとして残す（完全一致依存を主経路から外す）。
2. **Ctrl+X 単体対応**: 選択0〜1件でもカーソル行（またはサブツリー）を構造カット可能に（現行の `>=2` 制限を撤廃。1件時はカーソル行＋サブツリー）。
3. **Ctrl+A（アウトライン全選択）**: `olKeyDown` に追加。1回目=可視ノード全選択、2回目（または通常）=ネイティブ（行内テキスト）にフォールバック。実装は「contenteditable内テキスト選択」と競合しないよう、空/単一ノード時の挙動を定義。
4. **paste ハンドラ統合**: inline `onpaste` と `olSetupPasteHandler` を**1つの統合ハンドラ**に集約（画像→テキスト/HTML→構造の判定を一本化）。二重発火・順序依存を解消。
5. **親子保持**: 構造貼付で `parentId` を `_rel` から復元。
6. **回帰防止**: 既存の Shift 選択・Esc 解除・undo（`olPushHistory`）を維持。

**受入基準（Phase 1）**
- 自アプリ内で複数行（TODO含む・画像含む・親子含む）をコピー→別日付へ貼付して**無損失**（チェック状態・画像・親子が保持）。
- コピー後に他アプリでコピーし直しても、自アプリ貼付は **text/html マーカー**経由で構造復元（または素直にプレーン）—「貼れない」事象が起きない。
- Ctrl+X 単体で1行＋サブツリーがカットできる。
- Ctrl+A でアウトライン全行が選択され、Ctrl+C/X が効く。

### Phase 2 — リッチ貼付（HTML）　★実装済 v1.10.0-05301915-clip-p2（2026-05-30）
> 実装メモ: `olFilterStyle`/`olSanitizeFragment`(ホワイトリスト`OL_ALLOWED_TAGS`・危険要素/属性除去)/`olHtmlIsRich`/`olParseHtmlToNodes`(walkList で ul/ol 入れ子→indent、h→`<strong>`、表→1ノード、インライン装飾はnode.html保持)を新設。`olContainerPaste`①.5で「リッチかつ複数/構造」のみ介入、単一インラインはネイティブ委譲。外部imgは除去。ブラウザ実機検証済（サニタイズ/入れ子3階層/装飾保持/プレーン回帰なし）。

- マーカー無 `text/html` を **サニタイズ後**にノード化。
- **サニタイズ必須（重要）**: `node.html` はそのまま描画されるため、許可タグ/属性のホワイトリストで浄化（`script`/`style`/`on*`/`javascript:` 等を除去）。`<b><strong><i><em><u><a href><br><span style(色のみ)><ul><ol><li><p>` 等を許可。
- マッピング: `ul/ol>li` の入れ子→indent、`<p>/<div>`→行、`<b>/<strong>`→bold、`<a>`→リンク、テキスト色→`color`。
- 出典: Word / Google Docs / Web ページ / Notion 等からの貼付で構造・装飾を保持。

### Phase 3 — 表貼付
- **TSV/CSV**（Excel/Sheets）: `text/plain` がタブ区切り複数行 かつ `text/html` に `<table>` が無い場合、表として検出 → `node.html` に `<table>` を生成（`olInsertTable` の生成様式に合わせる）。
- **HTML表**: `text/html` に `<table>` → サニタイズして表ノード化。
- コピー時: 表を含むノードは `text/html` に `<table>` を、`text/plain` に TSV を書き出し（Excel等へ貼れる）。
- 代替案（要判断）: 表を1ノードの生html にするか、行ごとにノード展開するか。既定は **表ノード（生html）**。

### Phase 4 — 画像強化
- コピー時に画像（`node.html` 内 `<img>` / `node.images`）も `text/html` マーカーに含め、自アプリ間の往復で画像保持。
- 画像保存は **GitHub 必須を維持**（未設定時は従来どおりエラートースト）。
- 外部からの画像貼付は現行 `olSetupPasteHandler` を統合ハンドラに取り込み。

## 5. リスク / 運用上の懸念
- **HTMLサニタイズの徹底**（XSS/構造汚染）。Phase 2 の前提。
- **データ肥大**: リッチhtml・画像で `data.json`（既に約2MB）が増える。
- **クリップボードAPI**: `ClipboardItem` 書込は secure context 必須。フォールバック（`_olMultiClipboard`＋`execCommand`）を残す。
- **後方互換**: 既存ノード（プレーン）はそのまま。新フィールドは増やさない（クリップボードのみ拡張）。
- **二重発火**: ハンドラ統合まではPhase1で必ず解消（順序依存バグ防止）。

## 6. テスト項目（抜粋）
1. 自アプリ内: 複数行（TODO/画像/親子/色）コピー→別日付貼付→無損失。
2. 自アプリ内: カット→貼付で移動（元から消え、貼付先に復元）。
3. 他アプリ→自アプリ: Google Docs の箇条書き貼付→入れ子保持（P2）。
4. Excel→自アプリ: 範囲コピー→表になる（P3）。
5. 自アプリ→Excel: 表ノードコピー→Excelにセル展開（P3）。
6. スクショ貼付（GitHub設定済）→画像挿入（P4）/未設定→エラートースト。
7. Ctrl+A→全選択→Ctrl+C→外部に整形テキスト。
8. コピー後に他アプリで別コピー→自アプリ貼付が破綻しない（堅牢化の核心）。
9. 既存データの表示・undo・Shift選択に回帰が無い。
